# MVP 页面低保真线框图

> 本文档用于把 UX / 信息架构转成后续 UI 设计可直接承接的低保真页面草图。
> 当前阶段不定义视觉风格、不写代码、不决定组件库，只固定页面结构、信息优先级、状态位和主次动作。

---

## 1. 低保真约定

### 1.1 页面层级约定

- 每个页面同一时间只保留一个最主要动作。
- 四条路线不是首页工具入口，而是用户当下问题入口。
- 今日行动始终比报告、分析、解释更重要。
- 记录与复盘围绕“我今天实际做了什么”展开，不做复杂项目管理。
- 缺信息是正常状态，页面应转向补信息行动，而不是强行生成结论。

### 1.2 状态展示约定

- AI 参与时显示实时状态，但只显示用户能理解的产品语言。
- 不展示模型名、API key、prompt、token、fallback、重试次数、代码、异常堆栈等内部信息。
- 用户内容自动保存时，只显示“已保存 / 正在保存 / 保存失败，请重试”等状态。
- AI 失败时提供可继续的下一步，不把错误原因暴露给用户。

### 1.3 线框符号

```text
[ 页面标题 ]        页面主要识别信息
{ 状态提示 }        系统 / AI / 保存状态
( 输入区 )          用户填写区域
< 主要按钮 >        当前页面唯一主动作
< 次要按钮 >        可选动作
---                 页面区块分隔
```

## 2. MVP 整体页面流

```text
首页 / 今日入口
  |
  |-- 选择当下问题
  v
四路线轻输入页
  |
  |-- 信息足够 ----------------------\
  |                                  v
  |                           路线输出 + 今日行动卡
  |                                  |
  |                                  v
  |                           路线定制记录页
  |                                  |
  |                                  v
  |                           即时轻复盘页
  |
  |-- 信息不足
  v
缺信息行动页
  |
  v
保存补信息记录
  |
  v
重新判断信息是否足够

任意已保存状态
  |
  v
我的求职轨迹页
  |
  v
7天轻复盘页
```

## 3. 首页 / 今日入口

### 3.1 首次用户

```text
[ 21天陪跑 · 第1天 ]

不用一次想清楚，今天先推进一件事。

你现在最想先解决哪件事？

先选你今天最卡的一件事，后面只会给一个小行动。

[ 我不知道能投哪些岗位        > ]
[ 我的经历不知道怎么写进简历  > ]
[ 我看到岗位了，不知道投递前怎么改 > ]
[ 我投了一些，但没什么反馈    > ]

---
今天只需要完成一个 15-30 分钟的小行动。
```

### 3.2 回访用户

```text
[ 21天陪跑 · 第N天 ]

不用一次想清楚，今天先推进一件事。

{ 最近推进：已保存 1 条记录 / 暂无记录 }

[ 今日建议继续 ]
上次行动后的下一步：
补齐一个目标岗位的 JD 信息

< 继续今天的行动 >
< 换一个当前问题 >

---
[ 我的求职轨迹 > ]
```

### 3.3 回访用户：上次行动未完成

```text
[ 21天陪跑 · 第N天 ]

不用一次想清楚，今天先推进一件事。

{ 上次这一步还没完成，今天可以把它缩小一点。 }

[ 更小的一步 ]
先保存 1 个真实 JD / 先补 1 条投递记录 / 先写 1 段经历事实

< 继续一个更小版本 >
< 换一个当前问题 >
```

### 3.4 首页边界

- 首页不展示复杂数据面板。
- 首页不展示四路线的完整解释。
- 首页不放“测评”“生成报告”“匹配岗位”等强承诺入口。
- 首页核心任务是让用户选中一个当下问题，并进入当天行动。

## 4. 四路线轻输入页

### 4.1 通用结构

```text
[ 路线名称 ]
一句话说明当前页面要帮用户推进什么。

{ 已保存 / 正在保存 / 保存失败，请重试 }

( 输入区 1 )
( 输入区 2 )
( 输入区 3，可选 )

[ ] 我暂时不确定 / 现在没有更多信息

---
不用填完美。信息不够时，会先生成一个补信息行动。

< 生成今天先做的一步 >
< 先保存，稍后继续 >
```

提交后：

```text
{ 正在阅读你提供的信息... }
{ 正在整理今天可以先做的一步... }
{ 正在检查是否需要先补信息... }
```

### 4.2 路线一：方向 -> 岗位样本

```text
[ 我不知道能投哪些岗位 ]

( 你学过 / 做过 / 感兴趣的内容 )
( 你不想做或明显排斥的方向 )
( 你目前能接受的城市 / 行业 / 工作方式，可选 )

< 生成今天先做的一步 >
```

输出重点：

- 2-3 个搜索方向或关键词。
- 每个方向只说明为什么值得搜、先观察什么。
- 真实岗位样本需要用户自己去招聘平台或公开渠道保存。
- 不判断“适合 / 不适合”，不承诺“能投”。

### 4.3 路线二：经历 -> 简历材料

```text
[ 我的经历不知道怎么写进简历 ]

( 一段经历：课程 / 实习 / 项目 / 社团 / 兼职均可 )
( 你当时具体做了什么 )
( 有无结果、数据、作品或反馈，可选 )

< 生成今天先整理的一段简历材料 >
```

输出重点：

- 一段可整理的经历素材。
- 提醒用户补动作、对象、结果。
- 不包装成夸大履历，不替用户虚构成果。

### 4.4 路线三：JD -> 投递前最小修改

```text
[ 我看到岗位了，不知道投递前怎么改 ]

( 粘贴岗位 JD / 岗位要求 )
( 你准备投递的简历片段或经历 )
( 你最担心的问题，可选 )

< 生成投递前最小修改动作 >
```

输出重点：

- 1 个投递前最小修改动作。
- 可包含需要补齐的一处简历表述。
- 不输出匹配百分比，不说“能过 / 不能过”。

### 4.5 路线四：投递记录 -> 轻复盘

```text
[ 我投了一些，但没什么反馈 ]

( 投递记录 1：岗位 / 公司或平台 / 投递时间 / 反馈状态 )
( 投递记录 2，可选：岗位 / 公司或平台 / 投递时间 / 反馈状态 )
( JD 摘要 / 材料版本 / 怀疑点，可选 )

< 生成今天先复盘的一步 >
```

输出重点：

- 先允许补 1 条最低字段投递记录。
- 记录足够时，帮用户识别一个可调整点。
- 生成下一次投递前的小动作。
- 不猜测公司为什么没反馈，不做失败归因。

## 5. AI 处理中状态

### 5.1 页面内状态

AI 状态优先以内联提示出现，不单独制造复杂等待页。

```text
{ 正在阅读你提供的信息... }
{ 正在整理今天可以先做的一步... }
{ 正在检查是否需要先补信息... }
```

### 5.2 长等待状态

```text
[ 正在为你整理 ]

我们正在根据你刚才提供的信息，整理今天可以先推进的一步。

{ 这一步可能需要一点时间 }
{ 你可以先离开，当前内容已保存，下次回来会从这里继续。 }

< 先保存，稍后回来 >
```

### 5.3 失败状态

```text
[ 暂时没有整理成功 ]

你刚才填写的内容已经保存。
现在可以先继续补充信息，或稍后再试。

< 回到输入继续补充 >
< 稍后再试 >
```

### 5.4 保存失败状态

```text
[ 这次没有保存成功 ]

请先不要关闭页面，稍后再试。

< 重新保存 >
```

AI 失败和保存失败必须分开表达：

- AI 失败：内容已经保存，可以稍后继续整理。
- 保存失败：提醒用户不要关闭页面，并提供重新保存。

失败状态禁止展示：

- 模型名称。
- API key。
- prompt。
- token。
- fallback。
- 代码。
- 错误堆栈。
- 供应商返回原文。

## 6. 路线输出 + 今日行动卡

### 6.1 通用结构

```text
[ 今天先推进这一步 ]

{ 基于你刚才提供的信息整理 }

[ 简短判断 ]
现在最值得先做的是：把信息推进到可以行动的程度。

---
[ 路线输出区 ]
1-3 条，保持短。
只服务于今天这一步。

---
[ 今日行动 ]
行动：________
预计时间：15-30 分钟
完成标准：________
需要材料：________
做完后记录：________

< 我做完了，记录结果 >
< 先保存，稍后回来 >
```

### 6.2 路线输出区差异

路线一：

```text
[ 可先使用的搜索关键词 ]
关键词 A：先看工作内容和任职要求
关键词 B：先看是否反复出现同类要求
今天自己保存 1-3 个真实岗位
```

路线二：

```text
[ 今天先整理的一段经历 ]
经历主题：________
建议先补：动作 / 对象 / 结果
可先写成：一段朴素事实描述
```

路线三：

```text
[ 投递前最小修改 ]
先改这一处：________
为什么先改：与 JD 中的一个要求更接近
不要同时大改整份简历
```

路线四：

```text
[ 今天先复盘的一处投递 ]
先看这一类岗位：________
先检查一个环节：JD 对齐 / 简历表达 / 投递渠道 / 后续记录
```

### 6.3 今日行动卡边界

- 只能有一个行动。
- 不生成一长串任务清单。
- 不把行动变成“继续思考职业方向”这类不可验收任务。
- 不用“全面优化”“系统复盘”“深度分析”作为行动。
- 必须能在当天 15-30 分钟内完成或推进。

## 7. 缺信息行动页

```text
[ 先补一小块信息 ]

现在的信息还不足以整理出可靠建议。
这不是问题，今天先补一小块就够了。

---
[ 已经知道的 ]
- ________
- ________

[ 还缺的关键信息 ]
- ________

为什么只补这一项：
补完这一项后，系统才能判断 ________。

---
[ 今日行动 ]
行动：找到并保存 1 个真实 JD / 补写一段经历事实 / 先补 1 条投递记录
预计时间：15-30 分钟
完成标准：有一个可回填的信息

< 我补完了，去记录 >
< 先保存，稍后回来 >
```

边界：

- 不因为信息不足而编造建议。
- 不把缺信息包装成用户失败。
- 不要求用户一次补全所有资料。

## 8. 路线定制记录页

### 8.1 通用结构

```text
[ 记录今天做完了什么 ]

{ 已保存 / 正在保存 / 保存失败，请重试 }

[ 今天的行动 ]
系统带入今日行动内容

( 实际完成了什么 )
( 遇到什么卡点 )
( 有没有产生下一步线索 )

[ ] 我确认这条记录反映了我实际做过的事

< 保存并轻复盘 >
< 只保存 >
```

### 8.2 路线字段差异

路线一记录：

```text
( 今天看了哪个岗位样本 )
( 发现了哪些反复出现的要求 )
( 这个岗位让我更想了解 / 更排斥什么 )
```

路线二记录：

```text
( 今天整理了哪段经历 )
( 补齐了动作 / 对象 / 结果中的哪一项 )
( 还有哪一处不确定 )
```

路线三记录：

```text
( 今天投递前改了哪一处 )
( 修改依据来自 JD 的哪一点 )
( 是否已经投递 )
```

路线四记录：

```text
( 今天补了哪 1 条投递记录 )
( 如果已有多条：今天复盘了哪些投递 )
( 发现一个可能调整点 )
( 下一次投递前准备怎么试 )
```

### 8.3 AI 生成内容确认

当记录页带入 AI 整理出的内容时，必须让用户确认或编辑。

```text
{ 以下内容由系统根据你刚才的信息整理，请确认是否符合实际 }

( 可编辑内容 )

[ ] 我已检查，内容没有夸大或虚构
```

```text
[ 今天的小复盘 ]

[ 复盘依据 ]
你今天记录了：________

---
[ 看到的线索 ]
- ________
- ________

[ 还缺的信息 ]
- ________

[ 下一步建议 ]
明天可以先做：________

< 设为下一次行动 >
< 查看我的求职轨迹 >
```

边界：

- 不打分。
- 不做胜负判断。
- 不给“你适合什么”的结论。
- 不把一次行动上升为长期职业判断。

## 10. 我的求职轨迹页

```text
[ 我的求职轨迹 ]

这里仅展示你自己保存的求职记录。

[ 轻进度 ]
已记录行动：N 次
最近一次行动：________
最近一次复盘：________

---
[ 最近记录 ]
日期 / 路线 / 行动 / 结果
日期 / 路线 / 行动 / 结果

---
[ 分类查看 ]
[ 岗位样本 ]
[ 经历素材 ]
[ 投递记录 ]
[ 反馈与复盘 ]

< 回到今天的行动 >
```

记录项应提供：

- 编辑。
- 删除。

轨迹页应提供“清空我的记录”入口，但不放在主操作位置。

边界：

- 不是 CRM。
- 不做复杂漏斗。
- 不做排名、积分、连续打卡压力。
- 不把轨迹页变成数据驾驶舱。

- 岗位关键词 / 经历短板 / 投递卡点

[ 仍然缺的信息 ]
- ________

[ 下一轮先做一件事 ]
行动：________
预计时间：15-30 分钟

- 不输出周报式长报告。

- 不要求用户连续 7 天都有记录才可查看。

## 12. 保存后稍后继续

```text
[ 已保存 ]

你刚才填写的内容已经保存。
下次回来可以从这里继续。

< 回到首页 >
< 继续填写 >
```

适用场景：

- 输入页中途退出。
- AI 处理等待较久。
- 生成后未完成行动。
- 记录页只保存不复盘。

## 13. 页面禁止项

所有页面禁止展示：

- API key、密钥、环境变量。
- 代码、接口参数、数据库字段。
- prompt、token、上下文窗口。
- DeepSeek、Qwen、主模型、副模型、fallback、retry 等内部调度信息。
- 错误堆栈、供应商原始报错。
- “匹配度 xx%”“录取概率”“一定能投”“不能投”“包过”“保 offer”等表达。
- 未经用户确认的夸大经历、虚构结果、虚构项目成果。
- 暗示系统知道公司真实筛选原因的文案。

## 14. 交给 UI 阶段的约束

UI 阶段可以决定：

- 视觉风格。
- 字体层级。
- 颜色系统。
- 图标选择。
- 间距和响应式布局。
- 表单控件细节。

UI 阶段必须保留：

- 首页第一屏聚焦“今天先推进一件事”。
- 四路线作为问题入口，而不是工具集合。
- 今日行动为页面最高优先级。
- 每页一个主按钮。
- AI 状态可见但不暴露内部信息。
- 隐私与内部工程信息不出现在用户界面。
- 缺信息时转向补信息行动。
- 记录必须以用户实际完成为准。
- 复盘保持轻量，不做复杂评估系统。
- 低保真线框是页面结构草图，具体字段以《MVP 四路线输入输出数据设计.md》为准。
