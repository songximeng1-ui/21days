# MVP UX 准则

> 本文档基于《21天总准则》《新MVP PRD》《项目协作准则》《MVP PM决策共识》和《MVP UX信息架构第一版》整理。
> 后续 UX、UI、研发、AI 工作流、测试 agent 在开始相关工作前，应先阅读本文档。

## 1. 当前阶段边界

当前项目处于全新 MVP 的 UX / 信息架构阶段。

本阶段允许做：

- 用户流程图。
- 页面清单。
- 信息架构。
- 页面状态设计。
- 输入、输出、记录和复盘的结构设计。
- AI 工作流与页面状态之间的衔接设计。

本阶段不做：

- 代码实现。
- 旧代码复用判断。
- 视觉 UI 细节定稿。
- 完整简历编辑器。
- 招聘平台、岗位数据库、自动投递或自动抓取 JD。
- 通用自由问 AI。

## 2. 第一版 UX 总结论

第一版 MVP 采用：

```text
一个主流程 + 四条路线分支 + 一个轻量求职轨迹入口
```

核心判断：

- 第一版不是四个 AI 小工具集合。
- 首页不直接呈现四个并列工具入口。
- 四条路线是状态问题后的行动路径，不是彼此独立的产品模块。
- 今日行动卡是第一版 MVP 的核心对象。
- 每次使用都必须落到一个今日行动、一个真实记录或一次轻复盘。

第一版主流程必须始终服务：

```text
行动 -> 记录 -> 复盘 -> 调整
```

## 3. 首页准则

首页只回答一个核心问题：

```text
今天先从哪件事开始？
```

首页首屏应采用状态问题型体验，而不是营销页、功能入口页或材料输入页。

PM 已确认首页主文案：

```text
不用一次想清楚，今天先推进一件事。
```

PM 已确认承接问题：

```text
你现在最想先解决哪件事？
```

首页四个选择采用问题型表达：

- 我不知道能投哪些岗位。
- 我的经历不知道怎么写进简历。
- 我看到岗位了，不知道投递前怎么改。
- 我投了一些，但没什么反馈。

首页可以轻量显示：

```text
21 天陪跑 · 第 N 天
```

但首页主信息必须是今日行动或继续上次推进，而不是课程进度、打卡进度或长期规划。

## 4. 唯一主流程准则

第一版采用唯一主流程：

```text
首次进入
-> 选择当前最想解决的问题
-> 路线定制轻问答
-> AI 生成路线定制输出 + 今日行动卡
-> 用户完成行动并留下路线定制记录
-> 即时轻复盘
-> 下一步行动
-> 7 天阶段复盘
-> 21 天行动轨迹
```

四条路线进入后，AI 不应重复做“用户属于哪类状态”的诊断。

路线内 AI 的职责是：

- 围绕该路线痛点完成具体转化。
- 判断信息是否足够。
- 在信息不足时生成补信息型今日行动。
- 在信息足够时生成路线定制输出。
- 把输出收束为 1 个今日行动。
- 在用户记录后做即时轻复盘。

第一版页面命名统一为：

- 首页。
- 路线轻输入页。
- 缺信息行动页。
- 路线输出 + 今日行动卡页。
- 路线定制记录页。
- 即时轻复盘页。
- 我的求职轨迹页。
- 7 天轻复盘页。

不单独设置“状态选择页”“路线输出页”“今日行动卡页”或“21 天进度页”作为主页面。首页问题选择区嵌在首页；21 天周期感合并进“我的求职轨迹”的轻量头部。

## 4.1 首次进入与路线轻输入细化

第一版首次进入采用一屏四选一，不先追问诊断题。

首页先让用户回答：

```text
你现在最想先解决哪件事？
```

用户选择路线后，才进入路线定制轻输入。

路线轻输入允许追问 1-2 个必要问题，但只能围绕当前路线完成今日行动所需的信息展开。

例如：

- 选择“我看到岗位了，不知道投递前怎么改”后，可以要求用户贴真实 JD 和相关材料片段。
- 选择“我的经历不知道怎么写进简历”后，可以要求用户补一段真实经历和实际做过的动作。

不做：

- 首页前置多题诊断。
- 通用背景问卷。
- 路线内重复判断用户属于哪类状态。
- 为了生成更完整分析而延长首次输入。

第一版优先目标是让用户在 5 分钟内拿到今日行动卡。

这里的一屏四选一属于轻量状态诊断，不是四个并列工具入口。

用户选择的是“我现在最想先解决的问题”，不是“我要使用哪个 AI 工具”。

首页四个入口上方应固定给出轻提示：

```text
先选你今天最卡的一件事，后面只会给一个小行动。
```

首页入口、按钮和状态文案不得出现“工具”“功能”“生成报告”“岗位匹配”等暗示。四条路线的用户可见表达必须围绕“当前卡点”和“今天先做一步”。

## 4.2 路线切换与路线内追问边界

第一版允许用户切换路线，但应弱化为“换一个问题开始”，不应鼓励用户在多个路线之间反复跳转。

允许：

- 用户从首页选择另一个问题开始。
- 用户在路线内补充说明。
- 用户围绕当前路线目标、今日行动、记录或轻复盘追问。

不允许：

- 做通用自由问 AI。
- 在路线中频繁推荐用户切换到别的路线。
- 让用户感觉自己在四个工具之间试来试去。
- 让路线内追问脱离当前行动闭环。

如果用户明显选错路线，产品可以用克制表达引导：

```text
你现在的问题可能更接近“我看到岗位了，不知道投递前怎么改”。要不要从那里继续？
```

但不应把这种引导做成强制跳转。

## 5. 四条路线准则

### 5.1 方向 -> 岗位样本

用户问题：

```text
我不知道能投哪些岗位。
```

路线目标：

```text
从用户背景和真实经历中，找到 1-3 个可继续验证的真实岗位样本。
```

首版今日行动：

```text
今天先用这些关键词，自己保存 1-3 个真实岗位。
```

路线一输出只能给搜索方向、搜索关键词和观察重点。真实岗位样本必须由用户自己去招聘平台或公开渠道保存，产品不得暗示系统已经推荐、匹配或替用户找到真实岗位。

禁止做成：

- 职业测评。
- 最适合职业推荐。
- 泛泛方向建议。
- 单凭兴趣推职业。

### 5.2 经历 -> 简历材料

用户问题：

```text
我的经历不知道怎么写进简历。
```

路线目标：

```text
整理一段真实经历，形成克制、可信、可用于简历的片段。
```

首版今日行动：

```text
今天先整理一段真实经历。
```

禁止做成：

- 完整简历生成器。
- 精美包装工具。
- 夸大职责或成果的改写器。
- 把“参与”写成“主导”，把“协助”写成“负责”。

### 5.3 JD -> 投递前最小修改

用户问题：

```text
我看到岗位了，不知道投递前怎么改。
```

路线目标：

```text
基于真实 JD 和用户材料，判断材料与 JD 的支撑关系，并给出投递前最小修改动作。
```

首版今日行动：

```text
今天先对照 JD 做 1-2 条投递前最小修改。
```

禁止做成：

- 匹配度评分。
- 录取概率预测。
- 绝对能投或不能投判断。
- 用户本人适合或不适合判断。

### 5.4 投递记录 -> 轻复盘

用户问题：

```text
我投了一些，但没什么反馈。
```

路线目标：

```text
从模糊的“投了没反馈”，转成可复盘的投递记录和下一轮验证动作。
```

首版今日行动：

```text
今天先补 1 条最低字段投递记录；如果已经有 2-3 条，再选择 1 条先复盘。
```

投递记录路线采用两层完成标准：

- 最低完成：先补 1 条投递记录，至少包含岗位、公司或平台、投递时间、反馈状态。
- 继续补齐：再补 JD 摘要、材料版本、怀疑点，并逐步扩展到 2-3 条记录。

只有当记录足以支撑判断时，才进入轻复盘；否则继续生成补记录型今日行动。

禁止做成：

- 失败归因。
- 海投建议。
- 招聘方原因猜测。
- 没有投递记录时的完整复盘报告。

## 6. 今日行动卡准则

今日行动卡采用强约束单行动。

必须遵守：

- 每次只给 1 个今天最应该优先完成的动作。
- 不提供 3-5 个并列任务让用户自行选择。
- 不在主行动旁边附加多个并列建议。
- 行动应落在用户伸展区，优先控制在 15-30 分钟内可以完成。
- 行动必须有真实推进，并且能留下记录。

今日行动卡固定回答：

1. 今天只做什么。
2. 为什么先做这一步。
3. 具体怎么做。
4. 预计需要多久。
5. 做完后记录什么。

路线输出页和今日行动卡第一版应合并呈现。

推荐页面结构：

```text
一句短判断：现在能看出什么 / 还缺什么
-> 今日行动卡
-> 做完后记录什么
-> 记录入口
```

不单独做“分析报告页”。路线输出只是为了帮助用户理解为什么今天先做这一步，不能成为页面主角。

记录入口放在行动卡下方，推荐表达：

```text
我做完了，记录结果
```

用户可以先离开，完成线下行动后再回来记录。产品不应强迫用户当场填写完整记录。

## 6.1 未完成行动处理准则

用户没有完成今日行动时，产品不应惩罚、评价或制造焦虑。

应优先判断：

- 行动是否过大。
- 行动是否过难。
- 行动是否过模糊。
- 用户是否缺少开始的第一步。

下一步应把行动调回伸展区。

可以采用：

- 缩短行动时间。
- 减少行动数量。
- 给出第一个搜索词。
- 给出第一句话模板。
- 先补一个事实。
- 先保存一个岗位样本。
- 承认用户已经完成的部分，并接着推进下一小步。

用户可见表达应类似：

```text
没关系，今天先把这一步缩小一点。
```

禁止：

- 连续打卡压力。
- 失败评价。
- “你没有坚持”之类的人格判断。
- 因未完成而强行进入下一步。

回访首页遇到未完成行动时，应默认提供“缩小一步”的继续方式：

```text
上次这一步还没完成，今天可以把它缩小一点。
```

推荐主按钮：

```text
继续一个更小版本
```

推荐次按钮：

```text
换一个当前问题
```

## 6.2 完成感准则

用户完成今日行动并保存记录后，应有轻完成感，但不做游戏化打卡。

推荐表达：

```text
这一步已经留下记录，可以用于下次复盘。
```

可以展示：

- 已完成 1 次真实推进。
- 已保存 1 条记录。
- 下一次可以从这里继续。

不做：

- 积分。
- 排行榜。
- 连续打卡压迫。
- 夸张庆祝。
- 课程完成率。

## 7. 缺信息状态准则

缺信息状态不是失败状态，而是第一版 MVP 的正常状态。

当信息不足时，产品不生成看似完整但没有依据的报告，而是进入补信息型今日行动。

缺信息页至少包含：

1. 现在还不能可靠判断什么。
2. 目前已经知道什么。
3. 为什么只需要先补这一项。
4. 今天先补什么。

推荐解释句式：

```text
补完这一项后，系统才能判断 ________。
```

缺信息行动完成后的闭环必须是：

```text
完成补信息行动
-> 保存补信息记录
-> 重新判断信息是否足够
-> 信息足够：生成路线输出 + 今日行动卡
-> 信息仍不足：继续给 1 个补信息行动
```

缺信息行动不能默认直接进入完整复盘。只有当用户完成的是可复盘的真实行动，并且已有足够记录支撑时，才进入即时轻复盘。

示例：

```text
现在还不能可靠判断这份 JD 和你的材料支撑关系，因为还缺真实 JD。
目前能看出你已经有一个目标岗位名称。
今天先补这份岗位的真实 JD 或 3-5 条岗位要求。
```

禁止：

- 用泛泛建议填满页面。
- 假装已经完成深度分析。
- 让用户觉得是自己填错了。
- 让用户一次补完整所有信息。

## 8. 记录准则

第一版保留记录，但不做统一的大记录中心，也不做完整求职 CRM。

记录入口、记录字段和记录时机按路线定制。

用户感知应是：

```text
我做完这一步，顺手留下以后能复盘的东西。
```

记录对象只保存三类内容：

- 用户真实提供的岗位、经历、JD、投递和反馈。
- AI 基于真实证据生成的简历片段、修改建议和轻复盘结论。
- 下一次继续推进时必须知道的信息。

每个记录对象都要能回答：

- 这个记录来自用户的哪个真实动作？
- 它能不能支撑后续轻复盘？
- 它能不能帮助生成下一步行动？

如果不能，就不应进入第一版。

## 8.1 草稿保存准则

第一版应支持自动保存草稿。

原因：

- 用户填写的是真实经历、JD、投递记录等高摩擦内容。
- 用户中途退出、切换页面或等待 AI 时，不应丢失已填写内容。
- 自动保存能降低用户记录压力。

用户可见表达应普通：

```text
已保存草稿。
```

或：

```text
当前内容已保存，稍后可以继续。
```

不显示：

- 本地存储。
- 数据库写入。
- 接口保存成功。
- 请求失败。
- 内部字段名。

## 8.2 AI 生成内容确认准则

AI 生成的简历片段、修改建议、复盘结论和下一步行动，不能默认等同于用户事实。

涉及用户材料表达时，必须让用户确认、修改或保存后，才进入求职轨迹。

尤其是：

- 简历片段版本。
- 修改后材料片段。
- JD 对照修改记录。
- 投递记录中的材料版本。

推荐用户可见表达：

```text
如果这段表达符合真实情况，可以保存；如果没有做过，不要保留。
```

不允许：

- AI 自动把未确认内容写入用户正式材料。
- AI 自动把建议当作用户已经完成的事实。
- AI 把推测内容写进记录对象。

## 8.3 复制与导出边界

第一版可以支持轻量复制，但不做完整导出。

可以复制：

- 克制简历片段。
- 今日行动。
- 轻复盘中的下一步行动。
- 用户自己保存的岗位要求摘要。

不做：

- 完整报告导出。
- 一键生成完整简历。
- 投递记录批量导出。
- 带内部字段、模型状态或调试信息的复制内容。

复制内容必须只包含用户可见内容，不得包含代码、API key、prompt、token、模型名、内部路由、接口名或调试信息。

## 9. 轻复盘准则

第一版轻复盘不做完整报告，也不能薄到像随口建议。

固定结构为：

1. 复盘依据：本次复盘基于哪些真实记录。
2. 看到的线索：从记录中提炼 1-3 个具体线索。
3. 信息缺口：当前还缺哪些真实信息。
4. 下一步行动：只给 1 个优先动作。

第一版优先保证即时轻复盘成立。

回访轻复盘和 7 天阶段复盘可以继续细化，但不得偏离：

```text
行动 -> 记录 -> 复盘 -> 调整
```

## 10. 我的求职轨迹准则

第一版可以轻做“我的求职轨迹”入口。

作用：

- 让用户查看已留下的岗位样本、经历片段、材料版本、投递记录、反馈记录和轻复盘。
- 帮助用户感知自己在 21 天内确实有推进。
- 为回访和阶段复盘提供上下文。

边界：

- 只查看和轻量继续。
- 不做复杂筛选。
- 不做完整 CRM。
- 不做招聘平台式岗位管理。
- 不做复杂统计看板。

第一版“我的求职轨迹”只做最近推进和分类查看。

建议展示：

- 最近一次行动。
- 最近一次记录。
- 最近一次轻复盘。
- 分类入口：岗位样本、经历片段、投递记录、反馈记录。

不做：

- 多条件筛选。
- 数据看板。
- 投递漏斗。
- 岗位管理系统。
- 打卡日历。

## 10.1 7 天阶段复盘准则

7 天阶段复盘第一版轻做，不做重页面。

当用户已经留下一定记录后，可以出现轻量阶段复盘：

```text
过去 7 天你留下了这些推进记录，下一轮先验证这件事。
```

7 天阶段复盘应包含：

- 过去 7 天完成过哪些真实行动。
- 留下了哪些记录。
- 目前能看到的 1-3 个线索。
- 下一轮只给 1 个优先行动。

不做：

- 周报式长报告。
- 分数。
- 排名。
- 打卡日历。
- 课程完成率。
- 复杂统计图表。

## 11. AI 调用准则

本节是后续 AI 工作流设计必须遵守的内部准则，不作为用户可见页面信息。

第一版 AI 调用模型策略：

- DeepSeek 为主模型。
- Qwen 为副模型。
- 正常情况下只调用 DeepSeek。
- 只有当 DeepSeek 主模型重试后仍失败时，才调用 Qwen 副模型。
- 不因路线不同默认切换模型。
- 不因用户输入不足默认切换模型。
- 不把模型名称、重试、兜底等内部机制暴露给终端用户。

当主模型输出不满足产品边界时，应优先做主模型重试或进入缺信息状态，而不是直接放宽输出标准。

副模型只解决主模型调用失败后的可用性问题，不改变产品判断、UX 结构和安全边界。

## 11.1 用户可见 AI 状态准则

凡是页面步骤需要调用 AI，第一版应显示用户可理解的实时 AI 状态。

目的：

- 让用户知道当前确实进入了 AI 处理过程。
- 降低等待时的不确定感。
- 让用户理解系统正在基于他提供的真实信息整理结果。

用户可见状态应使用普通表达，例如：

- 正在阅读你提供的信息。
- 正在整理当前能看出的线索。
- 正在检查还缺哪些真实信息。
- 正在生成今天先做的一步。
- 正在整理复盘依据。

禁止向终端用户暴露：

- DeepSeek。
- Qwen。
- 主模型。
- 副模型。
- 重试。
- 兜底。
- fallback。
- token。
- prompt。
- 模型调用失败。

当 AI 调用失败或超过合理等待时间时，用户可见表达应落到产品状态，而不是内部错误。

第一版默认状态阈值：

- 0-8 秒：显示页面内 AI 状态。
- 超过 8 秒：显示长等待状态，并提示内容已保存。
- 超过 30 秒仍无结果：进入友好失败状态，并保留草稿。

推荐表达：

```text
这次暂时没整理出来。你可以先保存当前填写内容，稍后继续。
```

长等待状态应固定补充：

```text
你可以先离开，当前内容已保存，下次回来会从这里继续。
```

或：

```text
现在还不能可靠整理出下一步。今天可以先补这一小块信息。
```

AI 状态只能解释“正在做什么”，不能制造神秘感、炫技感或权威感。

## 11.2 用户界面隐私与内部信息准则

终端用户界面不得显示任何代码、密钥、配置、日志或内部实现信息。

禁止在用户界面显示：

- API key。
- 环境变量。
- 请求头。
- 服务端日志。
- 错误堆栈。
- 代码片段。
- 数据库连接信息。
- 模型配置。
- prompt 原文。
- token 信息。
- 调试信息。
- 内部路由名、接口名或字段契约。
- 开发者、主人或系统侧的私密信息。

当出现内部错误、AI 调用失败、网络异常或配置异常时，用户界面只能显示产品化表达。

推荐表达：

```text
这次暂时没整理出来。你可以先保存当前填写内容，稍后继续。
```

或：

```text
当前内容已经保存。稍后可以从这里继续。
```

禁止表达：

```text
API key 无效。
模型调用失败。
接口报错。
服务器返回 500。
DeepSeek 请求失败，切换 Qwen。
```

这条准则优先级高于调试便利性。任何页面、toast、弹窗、错误页、加载状态、导出内容和用户可复制内容都必须遵守。

同时，用户提供的真实经历、JD、投递记录、反馈记录和简历片段也属于用户隐私信息。

第一版用户界面不应把这些内容用于：

- 示例展示。
- 公共样例。
- 其他用户可见内容。
- 营销文案。
- 调试输出。

除非用户主动在当前页面查看、编辑、复制或保存，否则不应在无关页面暴露用户完整原文。

第一版必须提供最小隐私控制：

- 编辑单条记录。
- 删除单条记录。
- 清空我的记录。

如果第一版存在未登录或无账号状态，用户可见文案必须说明保存范围和可能丢失的边界，但不得暴露存储实现、数据库字段或接口信息。

“我的求职轨迹”页面建议显示轻提示：

```text
这里仅展示你自己保存的求职记录。
```

## 12. 用户可见文案准则

用户可见文案必须普通、具体、克制、可靠。

推荐表达：

- 今天先做这一步。
- 当前材料里还看不出来。
- 如果确实做过，可以补充；如果没有，不要编。
- 先找 1-3 个真实岗位样本验证。
- 先补 1 条最低字段投递记录；记录足够后再复盘。

避免表达：

- 你不适合。
- 你缺乏能力。
- 推荐职业。
- 匹配度。
- 录取概率。
- 大量投递提高机会。
- 系统兜底。
- 模型重试。
- fallback。

## 12.1 无障碍与包容性准则

第一版 UI 设计和研发必须保留基础可访问性：

- 四个问题入口不能只靠颜色区分。
- AI 状态、保存状态和错误状态不能只靠动效表达，必须有可读文本。
- 表单错误提示要说明具体可修正动作。
- 按钮文案应说明动作结果，避免只写“下一步”。
- 关键状态文本应能被屏幕阅读器读取。

推荐按钮表达：

```text
保存记录并轻复盘
```

避免只有：

```text
下一步
```

## 13. 第一版不做清单

第一版不做：

- 通用自由问 AI 入口。
- 四个并列 AI 工具入口。
- 完整求职 CRM。
- 完整简历编辑器。
- 招聘平台或岗位数据库。
- 自动抓取 JD。
- 自动投递。
- 职业测评。
- 课程表。
- 知识库或内容社区。
- 账号、支付、多用户后台、企业端、学校后台或家长端。
