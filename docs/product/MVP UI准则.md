# MVP UI 准则

> 本文档基于《21天总准则》《新MVP PRD》《项目协作准则》《MVP PM决策共识》《MVP UX信息架构第一版》《MVP UX准则》《MVP 页面低保真线框图》《MVP 四路线输入输出数据设计》和《MVP AI工作流与安全边界设计》整理。
> 后续 UI、前端、测试和其他 agent 在进入界面设计或实现前，应先阅读本文档。
> 本文档定义第一版 MVP 的 UI 设计方向、视觉准则、页面结构、组件边界、状态样式和用户可见文案语气，不进入代码实现。

## 1. UI 定位

第一版 MVP 的 UI 应被定义为：

```text
安静、可信、行动导向的求职行动陪跑界面。
```

它不是：

- 营销官网。
- AI 炫技页。
- 四个 AI 工具集合。
- 报告生成器。
- 招聘平台。
- 求职 CRM。
- 简历编辑器。
- 职业测评页。
- 课程或打卡产品。

用户进入产品后，首先应感到：

```text
我不用一次想清楚全部求职问题。
今天先完成一个具体的小行动就可以。
```

## 2. Apple-like 风格的使用边界

主人偏好类似苹果官网的克制、高级、清晰风格。第一版 UI 可以借鉴 Apple-like 的视觉原则，但不能照搬 Apple 官网的营销形态。

### 2.1 可以借鉴

- 克制的视觉层级：少装饰，把注意力集中在当前问题和今日行动卡。
- 大留白和低噪声：让普通本科应届生觉得可以开始，而不是被信息压住。
- 清晰短句：用自然、普通、低压的人话表达。
- 单一主任务：每屏只突出一个最重要动作。
- 高质量细节：圆角、边框、阴影、动效和图标都保持轻、稳、精确。
- 内容优先：界面不炫功能，把行动、记录、复盘做成核心内容。
- 稳定节奏：统一字号、间距和卡片密度，让产品显得可靠。
- 温和反馈：完成、保存、复盘时给轻微正反馈，不做夸张庆祝。

### 2.2 不应照搬

- 不做官网式大 Hero。
- 不做产品发布会式视觉冲击。
- 不做高冷奢侈感。
- 不把页面空到影响可用性。
- 不使用沉浸式大图、科技渐变、炫光或视差效果。
- 不写“重塑求职”“精准匹配”“提升成功率”等夸大文案。

最终方向应是：

```text
借 Apple-like 的克制、留白、清晰和细节质量；
不借 Apple 官网式营销表达、奢侈感和视觉冲击。
```

## 3. 视觉方向

### 3.1 整体气质

界面应像一个安静、认真、低压力的行动台面。

关键词：

- 安静。
- 清楚。
- 可信。
- 轻陪伴。
- 有记录感。
- 有下一步。

避免：

- 热闹。
- 炫技。
- 权威评判。
- 过度年轻化。
- 过度游戏化。
- 过度 SaaS 后台感。

## 4. 色彩系统

### 4.1 推荐色彩

第一版建议采用：

```text
暖白 / 浅灰底 + 柔和蓝绿行动色 + 少量低饱和琥珀提示色
```

推荐色值：

| 用途 | 色值 | 说明 |
|---|---|---|
| 页面背景 | `#F7F6F2` | 暖白，降低刺眼感 |
| 备用背景 | `#F8F8F5` | 极浅灰，可用于大面积页面底 |
| 内容面板 | `#FFFFFF` | 卡片与输入区 |
| 主文本 | `#1F2933` | 正文和重要标题 |
| 次级文本 | `#5F6B76` | 说明、辅助信息 |
| 弱提示 | `#8A949E` | 保存状态、轻提示 |
| 主行动色 | `#256C68` | 主按钮、关键选中态 |
| 备用主色 | `#2F5F72` | 更偏灰蓝的主色 |
| 行动卡底色 | `#E7F2EF` | 今日行动卡或选中背景 |
| 行动卡边线 | `#9BC7BD` | 强调但不刺眼 |
| 缺信息提示 | `#F4E8C8` | 低饱和琥珀 |
| 完成提示 | `#DDEFE5` | 低饱和绿色 |

### 4.2 禁止色彩倾向

第一版不建议使用：

- 大面积紫色或蓝紫渐变。
- 深色科技风。
- 高饱和亮蓝。
- 大面积金色、橙色或红色。
- 招聘平台式强红强蓝。
- 游戏化连续打卡色彩。

## 5. 字体与排版

### 5.1 字体

中文优先使用系统字体：

```text
PingFang SC / Microsoft YaHei / Noto Sans SC / system sans-serif
```

不建议首版引入强风格字体。字体应服务清晰阅读，而不是制造品牌个性。

### 5.2 字号层级

推荐层级：

| 层级 | 字号 | 使用场景 |
|---|---|---|
| 首页主文案 | 28-32px | 仅首页首屏使用 |
| 页面标题 | 22-24px | 路线页、记录页、复盘页 |
| 卡片标题 | 18px | 今日行动卡、复盘块 |
| 正文 | 15-16px | 主要说明和内容 |
| 辅助说明 | 13-14px | 状态、示例、提醒 |

标题不应过大到营销感。首页主文案可以有呼吸感，但首屏必须仍能看到问题选择入口。

### 5.3 文本节奏

- 短句优先。
- 每段只表达一个意思。
- 辅助说明不堆叠。
- 用户可见文案避免抽象术语。
- 按钮文案要说明动作结果，不只写“下一步”。

## 6. 布局与留白

### 6.1 移动端优先

第一版应移动端优先。

推荐宽度：

- 移动端：左右 16px 边距。
- 桌面端：主内容居中，宽度 560-680px。

不做：

- 多栏 dashboard。
- 大面积数据面板。
- 岗位管理表格。
- 复杂筛选布局。

### 6.2 留白原则

采用 Apple-like 的呼吸感，但不能空到像品牌页。

首页首屏必须尽量让用户看到：

- 主文案。
- 承接问题。
- 至少一部分问题选择卡。

路线输出页必须让用户尽快看到：

- 今日行动卡。
- 主按钮或记录入口。

## 7. 卡片与层级

### 7.1 今日行动卡

今日行动卡是第一版 MVP 的核心 UI 对象，永远是页面最高视觉优先级。

今日行动卡应包含：

1. 今天只做什么。
2. 为什么先做这一步。
3. 具体怎么做。
4. 预计需要多久。
5. 做完后记录什么。

样式建议：

- 使用白底或浅蓝绿底。
- 边框比普通卡片更明确。
- 圆角 8px 左右。
- 阴影极轻或不用阴影。
- 卡片内有清晰的时间、完成标准和记录入口。

### 7.2 普通信息卡

普通信息卡应弱于今日行动卡。

用于：

- 当前卡点。
- 路线输出摘要。
- 已知道的信息。
- 还缺的信息。
- 复盘依据。

不应让所有卡片同等突出。

### 7.3 问题选择卡

首页四个选择是“当前卡点”，不是“工具入口”。

样式建议：

- 白底。
- 浅灰边框。
- 选中时用主色边框、浅蓝绿底和左侧短强调线。
- 不使用大图标强化工具感。
- 不用“分析”“优化”“诊断”“匹配”等工具化词。

## 8. 图标、图片与动效

### 8.1 图标

图标可以用于辅助识别，但应低存在感。

适合图标：

- 行动。
- 记录。
- 复盘。
- 调整。
- 保存。
- 返回。

图标风格：

- 线性。
- 低饱和。
- 小尺寸。
- 不喧宾夺主。

### 8.2 图片

第一版不建议使用大主视觉图片。

允许：

- 空状态中的极轻量抽象图形。
- 记录轨迹中的极简视觉辅助。

不允许：

- 招聘平台式职场图片。
- AI 科技感插画。
- 营销官网大图。
- 夸张人物插画。

### 8.3 动效

只保留功能性微动效：

- 选择问题后的轻微状态变化。
- 保存成功的轻提示。
- 完成行动后的轻状态切换。
- 复盘结果淡入。

不做：

- 视差滚动。
- 炫光。
- 漂浮元素。
- 大面积 loading 动画。
- 打卡庆祝动画。

## 9. 情绪安抚型文案原则

第一版文案必须照顾普通本科应届生的情绪。

目标不是哄用户，而是让用户感到：

```text
我没有被评价。
我现在只是还缺一些真实信息。
今天先做一小步也算推进。
```

### 9.1 文案基本原则

优先使用：

- 正常化。
- 低压力。
- 具体行动。
- 真实依据。
- 下一步。

避免：

- 评判。
- 诊断感。
- 威胁感。
- 焦虑指标。
- 成功承诺。
- 权威口吻。

### 9.2 推荐表达

首页：

```text
不用一次想清楚，今天先推进一件事。
先选你今天最卡的一件事，后面只会给一个小行动。
```

缺信息：

```text
没关系，现在先不急着下结论。
还差一个关键信息，今天先补这一小块就可以继续。
```

经历整理：

```text
不用写得很漂亮，先把你确实做过的事说清楚。
如果没有明确结果，也可以直接写“无明确结果”。
```

JD 对照：

```text
这里先看材料和 JD 的支撑关系，不评价你本人适不适合。
```

投递无反馈：

```text
没有反馈不等于你不行。
我们先把真实投递记录补齐，下一次才好复盘。
```

未完成行动：

```text
没关系，可能是上一步太大了。
今天把它缩小一点，先完成一个更容易开始的版本。
```

完成记录：

```text
这一步已经留下记录，可以用于下次复盘。
```

### 9.3 禁止表达

禁止在用户界面中出现：

- 你不适合。
- 你缺乏能力。
- 你没有坚持。
- 失败原因是你不够匹配。
- 匹配度。
- 录取概率。
- 简历竞争力评分。
- 一键优化。
- 智能匹配。
- 提升成功率。
- 保 offer。
- 包过。

## 10. 页面结构准则

### 10.1 首页

首页只回答：

```text
今天先从哪件事开始？
```

结构：

1. 轻量顶部标识：`21 天陪跑 · 第 N 天`
2. 主文案：`不用一次想清楚，今天先推进一件事。`
3. 承接问题：`你现在最想先解决哪件事？`
4. 轻提示：`先选你今天最卡的一件事，后面只会给一个小行动。`
5. 四个问题型选择。
6. 一个主按钮。

禁止：

- 营销卖点。
- 功能宫格。
- 测评入口。
- 生成报告入口。
- 岗位匹配入口。

### 10.2 路线轻输入页

目标是收集少量真实信息，而不是做完整问卷。

结构：

1. 当前卡点复述。
2. 2-4 个轻问题。
3. 示例或“不确定”选项。
4. 草稿保存状态。
5. 主按钮：生成今天的行动。

### 10.3 缺信息行动页

缺信息页不是错误页。

结构：

1. 现在还不能可靠判断什么。
2. 目前已经知道什么。
3. 为什么只需要先补这一项。
4. 今天先补什么。
5. 主按钮。

### 10.4 路线输出 + 今日行动卡页

路线输出与今日行动卡合并展示。

页面重点顺序：

1. 一句短判断。
2. 路线输出摘要。
3. 今日行动卡。
4. 做完后记录什么。
5. 记录入口。

路线输出只服务今日行动，不做报告页。

### 10.5 路线定制记录页

记录页应让用户感觉：

```text
我做完这一步，顺手留下以后能复盘的东西。
```

结构：

1. 今日行动回顾。
2. 路线定制字段。
3. AI 整理内容确认。
4. 用户确认真实。
5. 主按钮：保存并轻复盘。

### 10.6 即时轻复盘页

固定四块：

1. 复盘依据。
2. 看到的线索。
3. 信息缺口。
4. 下一步行动。

下一步行动仍然只能有一个。

### 10.7 我的求职轨迹页

我的求职轨迹不是 CRM。

建议展示：

- 已记录行动数。
- 最近一次行动。
- 最近一次复盘。
- 最近记录时间线。
- 分类查看入口。

不做：

- 投递漏斗。
- 复杂筛选。
- 岗位池。
- 数据驾驶舱。
- 排名或分数。

### 10.8 7 天轻复盘页

7 天复盘不是周报，也不是成绩单。

结构：

1. 过去 7 天做过什么。
2. 留下了哪些记录。
3. 看到哪些线索。
4. 仍然缺什么信息。
5. 下一轮先做一件事。

## 11. 组件清单

### 11.1 基础组件

- 顶部轻导航。
- 第 N 天标签。
- 页面标题组。
- 问题选择卡。
- 单选项。
- 多选项。
- 短文本输入。
- 长文本输入。
- 主按钮。
- 次按钮。
- 文本链接。
- 状态提示条。
- 只读摘要块。
- 时间线记录项。

### 11.2 业务组件

- 当前卡点卡。
- 路线轻问答组。
- 今日行动卡。
- 缺信息行动卡。
- 路线输出摘要。
- 路线定制记录表单。
- AI 内容确认块。
- 即时轻复盘四块。
- 下一步行动卡。
- 求职轨迹时间线。
- 7 天复盘摘要。

### 11.3 首版不建议出现

- 聊天输入框。
- 多工具入口宫格。
- 复杂 dashboard。
- 岗位 CRM 表格。
- 匹配度图表。
- 分数、等级、排名。
- AI 报告下载。
- 大面积营销 hero。
- 模型运行日志。
- 课程日历。
- 连续打卡组件。

## 12. 状态样式

### 12.1 默认状态

- 背景干净。
- 主内容居中。
- 每页只有一个主按钮。
- 主按钮文案说明动作结果。

### 12.2 选择状态

未选中：

- 白底。
- 浅灰边框。

选中：

- 浅蓝绿底。
- 主色边框。
- 左侧短强调线。

四个问题入口不能只靠颜色区分。

### 12.3 加载状态

AI 加载状态必须使用用户能理解的普通表达。

可用：

- 正在阅读你提供的信息。
- 正在整理今天可以先做的一步。
- 正在检查还缺哪些真实信息。
- 正在整理复盘依据。

禁止：

- DeepSeek。
- Qwen。
- prompt。
- token。
- fallback。
- retry。
- 模型调用失败。
- API 报错。

### 12.4 缺信息状态

缺信息状态应使用温和提示色。

推荐文案：

```text
还缺一个关键信息，先补这一步就能继续。
```

缺信息状态必须提供一个可执行行动。

### 12.5 完成状态

完成状态应轻，不游戏化。

推荐文案：

```text
这一步已经留下记录，可以用于下次复盘。
```

不做：

- 积分。
- 排名。
- 连续打卡。
- 夸张庆祝。

### 12.6 错误状态

错误状态必须产品化表达。

AI 暂时失败：

```text
这次暂时没整理出来。你可以先保存当前填写内容，稍后继续。
```

保存失败：

```text
这次没有保存成功，请先不要关闭页面，稍后再试。
```

AI 失败和保存失败必须分开表达。

## 13. 可访问性准则

第一版必须保留基础可访问性：

- 颜色对比度至少满足 WCAG AA。
- 四个问题入口不能只靠颜色区分。
- AI 状态、保存状态和错误状态必须有可读文本。
- 表单错误提示要说明具体可修正动作。
- 按钮文案应说明动作结果。
- 关键状态文本应能被屏幕阅读器读取。
- 移动端点击区域不应过小。

## 14. UI 硬规则

后续 UI、前端、测试和其他 agent 必须遵守：

1. 今日行动卡永远是页面最高视觉优先级。
2. 每次只突出 1 个 15-30 分钟内可完成的行动。
3. 首页四个选择是当前卡点，不是四个工具。
4. 每页只能有一个主按钮。
5. 缺信息状态必须被设计成正常状态，并提供下一步。
6. 文案必须低压、具体、可执行。
7. 不使用成功率、录取概率、岗位匹配率、适合度评分等焦虑型指标。
8. 不使用大 Hero、科技渐变、AI 炫技视觉、招聘平台式信息流。
9. 不暴露模型名、prompt、token、API、fallback、错误堆栈或内部路由。
10. 移动端优先，首屏必须能看见问题选择或今日行动核心入口。
11. 记录页必须以用户实际完成为准，AI 生成内容必须经用户确认后才能进入正式记录。
12. 复盘保持轻量，固定围绕依据、线索、缺口和下一步行动。
