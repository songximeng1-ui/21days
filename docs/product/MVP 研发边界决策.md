# MVP 研发边界决策

> 本文档承接已确认的产品、UX、数据、AI 和 UI 准则，用于进入代码实现前固定首版研发边界。

## 1. 首版技术形态

首版采用 Next.js + React + TypeScript 单体应用。

原因：

- 当前仓库只有产品文档，没有旧代码需要继承。
- 第一版目标是快速验证行动、记录、轻复盘闭环，不需要微服务或前后端分仓。
- Next.js 可以同时承接移动端页面、API route、AI 编排接口和私测部署。

## 2. 隐私与存储边界

第一版不做账号系统。

为降低多人私测时的隐私风险，首版默认采用浏览器本地私有存储保存用户草稿和求职记录。

服务端首版只做：

- AI mock / 编排接口。
- 结构检查和安全检查。
- 不含用户完整原文的运行状态。

服务端首版不保存：

- 用户真实经历原文。
- JD 原文。
- 简历片段。
- 投递记录。
- 反馈记录。
- 用户怀疑的问题。

如果后续必须启用服务端持久化，必须先补充匿名设备级隔离、记录所有权校验、删除 / 清空能力和日志脱敏方案。

## 3. 四路线边界

首版采用一个 AI 编排器，但四条路线必须各自拥有独立策略。

编排器只负责：

- 接收路线请求。
- 调用当前路线策略。
- 执行结构检查。
- 执行安全检查。
- 处理 mock / 模型失败。
- 返回产品化状态。

四条路线策略分别负责：

- 输入足够标准。
- 缺信息规则。
- 路线定制输出结构。
- 今日行动生成。
- 记录提示字段。

禁止把四条路线揉成同一套报告模板。

## 4. AI 首版实现边界

首版先用 mock AI 跑通完整闭环。

mock AI 必须覆盖：

- 信息足够。
- 信息不足。
- 输出结构缺字段。
- 输出含禁词或违规判断。
- 主模型失败后副模型成功。
- 主副模型都失败。

真实 DeepSeek / Qwen 调用做成可插拔 provider，但没有 key 时不阻塞本地验证。

用户界面不得显示模型名、重试、兜底、fallback、prompt、token、接口名或错误堆栈。

## 5. 首版核心对象

首版只实现支撑闭环的最小对象：

- 今日行动实例。
- 岗位样本记录。
- 经历事实记录。
- 简历片段版本。
- JD 对照修改记录。
- 投递记录。
- 反馈记录。
- 轻复盘记录。
- 草稿。

不做岗位库、公司表、复杂标签、投递漏斗、看板、课程日历、完整 CRM 或完整简历编辑器。

## 6. 测试边界

首批测试必须覆盖：

- 四条路线完整输入时生成对应路线输出。
- 四条路线缺信息时进入补信息行动。
- 今日行动只能有一个。
- 今日行动在 15-30 分钟内可完成，并能留下记录。
- 安全检查拦截匹配度、录取概率、适合度、编造、海投和失败归因。
- AI 生成简历片段或修改后材料未经用户确认不能进入正式记录。
- 草稿保存和恢复。
- 编辑、删除、清空本地记录。
- 用户界面不暴露内部模型、prompt、token、fallback、接口名和错误堆栈。

## 7. 研发顺序

1. 建立 TypeScript 项目骨架。
2. 先写四路线规则和安全检查的失败测试。
3. 实现路线策略、今日行动结构和安全检查。
4. 实现本地草稿与记录存储。
5. 实现首页、路线输入、行动卡、记录、轻复盘和轨迹页。
6. 接入 mock AI 编排。
7. 跑通四路线端到端验证。
